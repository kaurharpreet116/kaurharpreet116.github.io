How I think as a Cloud Support Engineer

I troubleshoot with signals and scope first ‚Äî then mitigate safely ‚Äî then write the RCA so it doesn‚Äôt happen again.

Debug workflow

  1. Confirm impact (who/what/where, blast radius)
  2. Identify the failing layer (DNS ‚Üí network ‚Üí LB ‚Üí instance ‚Üí app)
  3. Check signals (metrics, logs, target health, flow signals)
  4. Mitigate (fast, safe rollback / bypass / restore)

Incident summary

Impact

  • Ops couldn‚Äôt access instance for live debugging
  • Users saw intermittent timeouts due to failed restarts

Symptoms

  • SSH timeout from bastion
  • HTTPS health checks failing
  • No recent CPU spike; instance status checks OK

Investigation steps

  1. Confirmed instance status checks OK (EC2 console / CLI).
  2. Verified route table + subnet association correct.
  3. Checked Security Group inbound rules for 22/443.
  4. Validated NACL rules weren‚Äôt blocking.
  5. Checked VPC Flow Logs for REJECT on target ENI.

Root cause

  • Re-added least-privilege inbound rules (22 from bastion SG, 443 from ALB SG).
  • Validated reachability and restored health checks.

  • Add change checklist: ‚ÄúValidate SG dependencies (ALB ‚Üí Instance, Bastion ‚Üí Instance)‚Äù

SEV-1 ALBTarget GroupHTTP 5xx MTTR: 19 min

Application down behind ALB ‚Äî Target health checks failing

ALB returned 502. Root cause was app listening on wrong port after deploy. Fixed + added health check guardrail.

Open incident write-up

Impact

  • All users received 502 Bad Gateway
  • Checkout API unavailable

Symptoms

  • ALB target group: unhealthy (health check failed)
  • ALB access logs show 502 spikes
  • Instance reachable internally, but app endpoint not responding

Timeline

  • T+0: Pager alert: 5xx error rate > threshold
  • T+5: Confirmed TG unhealthy; rolled back traffic shift paused
  • T+12: Found app listening on 8081 not 8080
  • T+19: Fixed service config; targets healthy

Investigation

  1. Checked ALB target health + reason codes.
  2. Validated security group path: ALB SG ‚Üí Instance SG on app port.
  3. SSH to instance; verified process and listening ports (ss -lntp).
  4. Checked app logs for bind/listen config.
  5. Verified health check path and matcher config.

Root cause

Deploy changed the service port to 8081, but ALB target group health checks were still configured for 8080.

Resolution

  • Restored correct app listen port (8080) via service config rollback.
  • Restarted service and verified health checks passing.

Prevention

  • Add CI check: ‚Äúservice port must match TG config‚Äù
  • Health check should validate a lightweight /health endpoint
  • Create rollback runbook + deploy guardrail for port changes

SEV-2 IAMKMSAccessDenied MTTR: 31 min

Production AccessDenied ‚Äî IAM policy missing KMS decrypt

App failed to read encrypted config. Debugged policy evaluation, fixed least-privilege permissions, documented prevention.

Open incident write-up

Impact

  • New deployments failed on startup (could not load config)
  • Increased error rate during scale events

Symptoms

  • CloudWatch logs: AccessDeniedException
  • Error mentions KMS decrypt / encrypted parameter

Investigation

  1. Confirmed failing call from logs (service + action + resource).
  2. Used IAM Policy Simulator / evaluation reasoning.
  3. Checked role trust policy (assume-role) correct.
  4. Verified key policy + IAM policy alignment.

Root cause

Instance role had permission to read the parameter/object, but was missing kms:Decrypt for the specific CMK.

Resolution

  • Added least-privilege kms:Decrypt scoped to the CMK ARN.
  • Validated new deployment succeeded; monitored error rate back to baseline.

Prevention

  • Standardize ‚Äúencrypted config access‚Äù policy module
  • Deployment pre-check: simulate role access to required resources
  • Alert on sudden AccessDenied spikes by service
